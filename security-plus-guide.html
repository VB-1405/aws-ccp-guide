1.1 Security Controls

Security controls are safeguards or countermeasures designed to avoid, detect, counteract, or minimize security risks. They are organized into categories (what they do) and types (how they operate).

⚠️ Exam trap: Know BOTH the category AND type of a control. A security guard is a Physical category control AND a Deterrent/Detective type. These overlap and questions test you on both dimensions.

Control Categories

CategoryDescriptionExamples
TechnicalControls implemented via technology/systemsFirewalls, antivirus, encryption, IDS/IPS, MFA
ManagerialAdministrative controls — policies, proceduresSecurity policies, risk assessments, background checks
OperationalDay-to-day operations by peopleSecurity awareness training, guard patrols, incident response
PhysicalReal-world barriers and protectionsLocks, fences, badge readers, cameras, bollards

Control Types

TypeGoalExample
PreventiveStop the incident before it happensFirewall rules, encryption, security guards
DeterrentDiscourage attackers from attemptingWarning signs, login banners, visible cameras
DetectiveIdentify that an incident occurred or is occurringIDS, SIEM, audit logs, motion sensors
CorrectiveLimit damage and restore after an incidentBackups, patch management, antivirus removal
CompensatingAlternative when primary control isn't feasibleIncreased monitoring when patching isn't possible
DirectiveDirect someone to follow a security policyAcceptable use policies, training requirements

1.2 Security Concepts

The CIA Triad

The CIA triad is the foundation of all IT security decisions. Every security control maps back to one or more of these three goals.

PrincipleMeaningThreat to itControl example
ConfidentialityOnly authorized parties can access dataEavesdropping, data breachEncryption, access controls, MFA
IntegrityData has not been altered without authorizationTampering, man-in-the-middleHashing, digital signatures, checksums
AvailabilitySystems/data are accessible when neededDDoS, ransomware, hardware failureRedundancy, backups, failover

Non-repudiation

Non-repudiation ensures that an action cannot be denied by the party who performed it. Achieved through digital signatures — the sender uses their private key to sign; the receiver verifies with the sender's public key. If the signature checks out, the sender cannot deny sending the message.

⚠️ Non-repudiation is a cryptography concept. Digital signatures provide: Authentication (who sent it) + Integrity (not altered) + Non-repudiation (can't deny). All three from one mechanism.

AAA — Authentication, Authorization, Accounting

ComponentQuestion answeredExample
AuthenticationWho are you? (Verify identity)Username + password, MFA, biometrics
AuthorizationWhat are you allowed to do?ACLs, role-based permissions, policies
AccountingWhat did you do? (Audit trail)Log files, SIEM, audit records

Gap Analysis

A gap analysis compares your current security posture against a desired baseline (like NIST CSF or CIS Controls). The "gap" is the difference between where you are and where you need to be. Used to prioritize remediation efforts and create a roadmap.

Zero Trust

Zero Trust is a security model based on the principle: "Never trust, always verify." Assumes breach is already possible — even inside the network perimeter. Every access request is authenticated, authorized, and continuously validated regardless of network location.

  • Control plane: Adaptive identity, threat scope reduction, policy-driven access control
  • Data plane: Subject/system + policy engine + policy enforcement point → access to resources

Physical Security

ControlPurpose
BollardsPrevent vehicle ramming attacks
Access control vestibule (mantrap)Prevent tailgating — only one person enters at a time
Fencing / cagesPhysical perimeter barriers
Video surveillance (CCTV)Detective and deterrent control
GuardsDeterrent, preventive, and corrective
Sensors (motion, infrared)Detective control — alert to presence

Deception & Disruption

  • Honeypot: Fake attractive system designed to lure attackers and study their techniques
  • Honeynet: Network of honeypots simulating an entire environment
  • Honeyfile: Fake file that alerts when accessed
  • Honeytokens: Fake credentials that trigger alerts when used
  • DNS sinkhole: Redirects known malicious domains to a controlled server

1.3 Change Management

Change management is the formal process for making controlled changes to IT systems, minimizing the risk of outages or security incidents caused by unplanned or unauthorized modifications.

Change Management Process

StepDescription
Change requestFormal request submitted with scope, reason, and impact
Approval (CAB)Change Advisory Board reviews and approves/denies
TestingChanges tested in sandbox/dev environment first
Backout planDefined rollback process if change fails
Maintenance windowScheduled low-impact time for deployment
DocumentationRecord what changed, when, and by whom

Technical Change Management Concepts

  • Allow list / Deny list: Allow list = only approved software runs. Deny list = block specific known-bad software
  • Restricted activities: Some changes require elevated approval (firewall rule changes, production deployments)
  • Version control: Track changes to code and configurations — enables rollback
  • Legacy applications: Harder to update; may require compensating controls instead of patches
  • Dependencies: Changes to one system may break others — document all dependencies before changing
⚠️ Exam scenario: "An organization needs to deploy a critical patch immediately." The correct approach still involves a formal change request, even if the timeline is compressed — emergency change process exists for this.

1.4 Cryptographic Solutions

Symmetric vs Asymmetric Encryption

FeatureSymmetricAsymmetric
KeysSame key encrypts and decryptsPublic key encrypts, private key decrypts
SpeedFast — good for bulk dataSlow — good for key exchange and signatures
Key sharing problemYes — how do you share the key securely?No — public key is freely shared
ExamplesAES, DES, 3DES, RC4RSA, ECC, Diffie-Hellman
Use caseEncrypting bulk data (files, drives)TLS handshake, digital signatures, key exchange

Key Exchange

Diffie-Hellman (DH): Allows two parties to establish a shared secret over an insecure channel without ever transmitting the secret itself. Used in TLS to establish a session key. Ephemeral DH (DHE) generates a new key each session — provides Perfect Forward Secrecy (PFS).

⚠️ Perfect Forward Secrecy (PFS): Even if the server's private key is compromised, past sessions cannot be decrypted because each session used a unique ephemeral key. DHE and ECDHE provide PFS.

Hashing

AlgorithmOutput SizeStatus
MD5128-bitBroken — collision attacks exist
SHA-1160-bitDeprecated — collision found 2017
SHA-256256-bitCurrent standard — secure
SHA-3VariableModern alternative to SHA-2
HMACDepends on hash usedHash + secret key = message authentication

PKI & Certificates

  • Certificate Authority (CA): Trusted third party that issues and signs digital certificates
  • Root CA → Intermediate CA → End-entity cert: Chain of trust — verify all levels
  • Certificate fields: Subject, Issuer, Serial number, Validity period, Public key, Digital signature
  • CRL: Certificate Revocation List — list of revoked certs (batch, can be outdated)
  • OCSP: Online Certificate Status Protocol — real-time cert validation (preferred)
  • Self-signed cert: Signed by itself — no third-party trust, used internally

Obfuscation Techniques

TechniqueHow It Works
SteganographyHides data inside another file (image, audio) — security through obscurity
TokenizationReplace sensitive data with non-sensitive token — actual data stored in vault
Data maskingReplace real data with fictional but realistic data — used in dev/test environments

Serverless, Microservices & Containers

  • Serverless: No server management — code runs on demand. Attack surface: function code and event triggers
  • Microservices: Application broken into small independent services. Each has its own attack surface — secure APIs between them
  • Containers (Docker/K8s): Share OS kernel — container escape is a real attack vector. Use read-only containers, non-root users

3.4 Data Protection

Data States

StateDescriptionProtection method
Data at restStored on disk, database, backup mediaFull disk encryption (FDE), file-level encryption, AES-256
Data in transitMoving across a networkTLS, HTTPS, IPSec VPN, SSH
Data in useBeing actively processed in memorySecure enclave, memory encryption, access controls

Data Classification

  • Public: No harm if disclosed — marketing materials, press releases
  • Private/Internal: Not for public — employee info, internal docs
  • Sensitive/Confidential: Serious harm if disclosed — trade secrets, PII
    • Critical/Top Secret: Severe harm — national security, financial data

Data Loss Prevention (DLP)

DLP tools identify, monitor, and protect sensitive data from unauthorized transfer or disclosure. Can operate at endpoint level (blocking USB transfers), network level (inspecting outbound traffic), or cloud level (CASB).

4.1 Identity & Access Management

Authentication Methods

FactorTypeExamples
Something you knowKnowledgePassword, PIN, security questions
Something you havePossessionSmart card, hardware token (TOTP), phone (push notification)
Something you areInherence (biometrics)Fingerprint, retina, facial recognition, voice
Somewhere you areLocationGPS location, IP geolocation
Something you doBehaviorTyping patterns, gait analysis

MFA & Passwordless

  • MFA: Two or more factors from different categories. Password + SMS code = two factors but SMS is weak (SIM swapping)
  • TOTP: Time-based One Time Password — generates new code every 30 seconds (Google Authenticator)
  • FIDO2/WebAuthn: Passwordless using hardware keys (YubiKey) or biometrics — phishing-resistant
  • SSO: Single Sign-On — authenticate once, access multiple apps. SAML, OAuth, OpenID Connect

Access Control Models

ModelHow It WorksUse Case
DACDiscretionary — owner controls access to their resourcesStandard file permissions (Linux chmod)
MACMandatory — system enforces based on labels (Top Secret, Secret)Government/military — SELinux
RBACRole-based — access granted based on job roleEnterprise IAM — most common
ABACAttribute-based — access based on user, resource, and environment attributesFine-grained cloud access policies
RuBACRule-based — access determined by system rulesFirewall ACLs, time-of-day restrictions

Privileged Access Management (PAM)

  • Just-in-time (JIT) provisioning — admin access granted only when needed, for limited time
  • Privileged access workstations (PAW) — dedicated hardened workstations for admin tasks
  • Password vaulting — credentials stored in encrypted vault, never seen by humans

Security Information and Event Management (SIEM) aggregates and correlates log data from across the environment to detect threats and generate alerts.

4.3 Incident Response

⚠️ The IR lifecycle is one of the most tested topics on Security+. Know all phases and their ORDER.

Incident Response Lifecycle (NIST SP 800-61)

PhaseActivities
1. PreparationPolicies, playbooks, tools, training — BEFORE an incident
2. Detection & AnalysisIdentify the incident — logs, alerts, SIEM, user reports. Determine scope and severity.
3. ContainmentStop the spread. Short-term (isolate) then long-term (patch, rebuild). Preserve evidence.
4. EradicationRemove malware, close attack vectors, patch vulnerabilities
5. RecoveryRestore systems, verify functionality, monitor for re-infection
6. Lessons LearnedPost-incident report — what happened, what worked, what to improve

Containment Strategies

  • Isolation: Disconnect compromised system from network immediately
  • Segmentation: Restrict lateral movement without fully disconnecting
  • Remove from service: Take offline for forensic analysis
  • Preserve evidence: Image the disk BEFORE any changes — chain of custody

Attack Frameworks

FrameworkWhat It Describes
MITRE ATT&CKAdversary tactics, techniques, and procedures (TTPs) — real observed attacker behaviors
Cyber Kill Chain7 stages of a cyberattack (Lockheed Martin) — Recon → Weaponize → Deliver → Exploit → Install → C2 → Actions
Diamond ModelAdversary, Infrastructure, Capability, Victim — analyze attacker campaigns

4.4 Digital Forensics

Forensics Principles

  • Chain of custody: Document every person who handles evidence — admissibility in court
  • Legal hold: Preserve evidence when litigation is anticipated — do NOT delete
  • Data integrity: Hash the evidence before and after — any change is detectable
  • Write blocker: Hardware device prevents any writes to evidence drive during imaging

Order of Volatility (Most → Least Volatile)

SourceVolatility
CPU registers & cacheMost volatile — lost on power-off
RAM (running processes, network connections)Very volatile
Swap/page fileVolatile
Network traffic / temp filesVolatile
Hard disk dataLess volatile
Remote logs, cloud storageLeast volatile
⚠️ Collect most volatile evidence FIRST. RAM has running malware, encryption keys, network connections — all lost if system is powered off. Disk image can wait; RAM cannot.

Forensics Techniques

  • Disk imaging: Bit-for-bit copy of entire drive including deleted files and slack space (dd, FTK Imager)
  • Memory acquisition: Capture RAM contents — reveals running processes, malware, encryption keys
  • Log analysis: Review system, application, security logs for timeline reconstruction
  • Steganography analysis: Detect hidden data in files
  • File carving: Recover deleted files from unallocated disk space

5.2 Risk Management

Risk Concepts

TermDefinition
VulnerabilityA weakness in a system
ThreatA potential danger that could exploit a vulnerability
RiskLikelihood × Impact of a threat exploiting a vulnerability
Threat actorThe entity that carries out the threat
ExposureThe degree to which an asset is exposed to threats

Risk Response Strategies

StrategyDescriptionExample
AcceptAcknowledge risk and do nothing — acceptable risk levelMinor vulnerability with negligible impact
AvoidStop the activity that creates the risk entirelyStop using a vulnerable system
TransferShift risk to another partyCyber insurance, outsourcing
MitigateReduce likelihood or impact with controlsPatch, firewall, MFA

Risk Quantification

  • SLE (Single Loss Expectancy) = Asset Value × Exposure Factor
  • ARO (Annualized Rate of Occurrence) = how often the event occurs per year
  • ALE (Annualized Loss Expectancy) = SLE × ARO — annual cost of the risk
  • Residual risk: Risk that remains after controls are applied
Example: Server worth $100,000. A fire would destroy 40% (EF=0.4). SLE = $40,000. Fire occurs once every 10 years (ARO=0.1). ALE = $40,000 × 0.1 = $4,000/year. If a sprinkler system costs $2,000/year — worth it.

5.3 Compliance, Regulations & Privacy

Key Regulations

RegulationScopeKey Requirement
GDPREU citizens' data — any company worldwideConsent required, right to erasure, 72-hr breach notification

Privacy Concepts

  • PII (Personally Identifiable Information): Data that can identify an individual — name, SSN, email, biometrics
  • PHI (Protected Health Information): Medical records, diagnoses, treatment history
  • Data minimization: Collect only what you need — reduces risk and compliance burden
  • Data retention: Keep data only as long as required — legally and operationally
  • Right to be forgotten: GDPR right — request deletion of personal data
  • Privacy impact assessment (PIA): Evaluate privacy risks before deploying new system
